DROID SURVAIR CLOUD PORTAL - DUAL-WORLD ENVIRONMENT

This machine contains two separate operating worlds for the same React + FastAPI portal.

LIVE CLOUDFLARE WORLD
- Frontend: Vite preview on http://localhost:4173
- Backend: FastAPI on http://127.0.0.1:8000
- Database: Project_Data\droid_cloud_prod.db
- External Live release folder: D:\1_Portal_Workflows_development\DroidSurvair_Live_Release
- Frontend bundle served from: D:\1_Portal_Workflows_development\DroidSurvair_Live_Release\frontend\dist_live
- Backend code served from: D:\1_Portal_Workflows_development\DroidSurvair_Live_Release\backend

- Cloudflare Tunnel should point to the frontend preview port 4173.
- DEV_MODE=False tells the backend to use droid_cloud_prod.db.

DEV CURSOR WORLD
- Frontend: Vite dev server on http://localhost:5173
- Backend: FastAPI on http://127.0.0.1:8001
- Database: PostgreSQL/PostGIS database droid_master_suite_dev
- Start script: Droid_Environment_Manager\1_Start_Dev_Environment.bat
- Setup script: Droid_Environment_Manager\4_Setup_Dev_PostgreSQL.bat
- DB_BACKEND=postgres tells the Dev backend to use droid_master_suite_dev.

CRUCIAL DATA SAFETY RULE
- NEVER copy Project_Data\droid_cloud_dev.db over Project_Data\droid_cloud_prod.db manually.
- droid_cloud_prod.db is the live client database.
- droid_cloud_dev.db is the preserved read-only source used for the completed Dev migration.
- droid_master_suite_dev is the active Dev database and must never be connected to Live.
- Copying dev DB back to live can overwrite real client data, users, sessions, spatial assignments, and activity logs.
- Dev startup does not copy or open the Live SQLite database.
- Deploy scripts copy code/builds to Live only; they must never copy Dev DB -> Live DB.
- Legacy issues.db may remain as an old backup/source file. Do not treat it as the active live database after migration.

DATABASE LOCK PROTECTION
- Live SQLite backend connections enable WAL mode:
  PRAGMA journal_mode=WAL;
  PRAGMA synchronous=NORMAL;
- WAL mode may create .db-wal and .db-shm files beside the selected database. This is expected.

UPDATES AND DEPLOYMENT
- Dev source edits are made in backend\ and frontend\src.
- Live does not serve normal frontend\dist anymore.
- Live serves only the external release bundle at D:\1_Portal_Workflows_development\DroidSurvair_Live_Release\frontend\dist_live.
- Live backend does not run directly from backend\ anymore.
- Live runs only from D:\1_Portal_Workflows_development\DroidSurvair_Live_Release\backend.
- 2_Start_Live_Portal.bat starts the already deployed Live bundle/release copy. It only creates them once if missing.
- 3_Nightly_Auto_Deploy.bat asks whether to deploy immediately or wait until the next 3:00 AM.
- Selecting Deploy Now immediately refreshes the external Live frontend bundle and backend release copy.
- Selecting Schedule waits until the next 3:00 AM before refreshing the Live release.
- Frontend checks /api/version and shows a toast when a newer backend/frontend version is available.
- The client does not hard-refresh during active work.
- Between 2:00 AM and 4:00 AM, the frontend may silently reload only after 30 minutes of user inactivity.
- Backend/frontend logic changes go live through Droid_Environment_Manager\3_Nightly_Auto_Deploy.bat.
- Manual Live restart only restarts the last deployed release copy; it should not pick up current dev source changes.
- Database schema changes go live when the deployed backend starts and runs ensure_tables against droid_cloud_prod.db.
- Dev testing data never moves to Live unless explicitly entered again in Live.

POINT CLOUD EPT CONVERSION
- The old PotreeConverter workflow is removed.
- Droid Cloud now converts LAS/LAZ point clouds to EPT and opens them in the Droid EPT viewer.
- Strict converter is Untwine only because it creates the proper chunked EPT hierarchy needed by the viewer.
- Entwine and PDAL writers.ept fallback are disabled for portal EPT generation.
- Current QGIS Untwine path:
  C:\Program Files\QGIS 3.22.8\apps\qgis-ltr\untwine.exe
- Portable converter files can be placed in Tools\EPT.
- Dev and Live backend startup scripts first check the QGIS Untwine path, then Tools\EPT and the system PATH.
- This machine also checks the isolated Miniforge environment at:
  C:\Users\droid\miniforge3\envs\droid-ept\Library\bin
- If Untwine is not found, point cloud conversion fails explicitly instead of silently falling back.
